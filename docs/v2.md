# Chalagente v2

v1 was a single-tenant proof: one WhatsApp linked device, one hardcoded script, no
accounts. v2 turns this into a real product: businesses sign up, link their own
WhatsApp number, paste in their info, and an agent fields customer messages on
their behalf.

## End-to-end story

1. A business lands on the marketing page and signs up.
2. They give us their business details (Google Maps + free-form extras).
3. They scan a QR to link their WhatsApp number to our agent.
4. We hand them their own "message me" QR pointing at their number, plus a test
   harness to confirm it works.
5. From then on, customers WhatsApp the business, the agent answers using the
   business's info and a small toolbox. The owner can flip the agent off any
   time from the dashboard.

## Auth

Email + magic link via [better-auth](https://better-auth.com). No phone
verification — the WhatsApp pairing step is the real proof of control over the
number, so a separate SMS/OTP flow is wasted effort.

Existing single-tenant session in `/data/store.db` gets migrated to belong to a
seed user (us). New signups go through the normal flow.

## Onboarding flow

After magic-link sign-in, the user walks through a short wizard. Each step
saves on submit so they can drop off and come back.

1. **Business basics.** Search Google Maps, pick the place, we pull
   name / address / hours / phone / categories / website. They can edit the
   pulled fields.
2. **Extra info.** Free-form textarea — pricing, FAQs, tone, anything the Maps
   listing doesn't cover. This is what differentiates a generic answer from a
   useful one.
3. **Link WhatsApp.** Per-user QR (same pattern as v1, just scoped to this
   account). On `success` we store the device and move on. The store needs to
   become multi-tenant — see Storage below.
4. **Test run.** We give them a `wa.me/<their-number>` link and QR. They send
   a couple of messages from another phone and watch the agent reply in a live
   transcript view. There's a "this works" button that finalises onboarding.

## Admin dashboard

One page per business, lives at `/app`. Tabs:

- **Conversations.** Live transcript of every chat, newest first. Same event
  bus we already have, just filtered per tenant.
- **Agent.** Big on/off toggle. When off, incoming messages get logged but no
  reply goes out. Also: model picker, system-prompt preview (read-only —
  derived from business info), temperature-ish knobs only if we actually need
  them.
- **Business info.** Same fields as onboarding, editable. Changes invalidate
  the cached system prompt.
- **Tools.** List of available tools (ElevenLabs transcribe, ElevenLabs TTS,
  Maps lookup, plus any custom Make/n8n webhooks the business wires up). Each
  has an enable toggle and per-tool config.
- **Share.** The customer-facing QR + `wa.me` link, copy buttons, downloadable
  PNG for printing.

## Agent

Per-tenant system prompt built from: business basics + extra info + the
current date + a short instruction block. Tools attached to the agent:

- Custom webhooks the owner adds (URL + JSON schema, called by the agent when
  it decides to).

### ElevenLabs (interface layer, not the agent's problem)

Two pieces, both bypass the agent's tool-calling — they run unconditionally on
the message envelope before/after the agent sees it.

**Inbound voice → text.** When a WhatsApp `audioMessage` comes in:

1. Download the encrypted `.ogg` (Opus) via whatsmeow.
2. POST to ElevenLabs `/v1/speech-to-text` with `model_id=scribe_v1`. Ask for
   language detection in the response.
3. Feed the transcript to the agent as the user turn, prefixed with
   `[voice note, lang=<detected>]` so the prompt knows it was spoken.
4. Stash the detected language on the conversation so the outbound side can
   match it.

**Outbound text → voice.** If the inbound message was a voice note, reply as
a voice note in the same language. Otherwise reply as text. (Owner can force
"always voice" or "always text" in the Agent tab.)

1. Take the agent's text output.
2. POST to ElevenLabs `/v1/text-to-speech/<voice_id>` with
   `model_id=eleven_multilingual_v2`, output format `mp3_44100_128`.
3. Voice per business, configurable in the Tools tab; default to a neutral
   multilingual voice. Don't pick per language — `eleven_multilingual_v2`
   speaks whatever language the input text is in with one voice.
4. Send via whatsmeow as an `audioMessage` with `PTT=true` so it shows up as a
   proper voice note, not an attachment.

Cache TTS by `(voice_id, text)` hash — the hardcoded-response audios from v1.6
should hit the cache and never call ElevenLabs twice.

Errors fall back to text: if STT fails, tell the agent we couldn't hear the
note and let it ask for a retype; if TTS fails, just send the text reply.

API key is a single `ELEVENLABS_API_KEY` env var for now — per-business keys
can wait until someone asks.

### Multimodal

Images and short videos from customers go to the agent as attachments
(vision-capable model). Outbound is text or voice only — no generated images.

## Storage

Single shared DB. whatsmeow's tables get a `tenant_id` column and we run one
`whatsmeow.Client` per linked business, all pointing at the same store. App
tables (users, businesses, tool config, agent toggle state, conversation logs)
live in the same DB.

SQLite for now (`/data/app.db`), Postgres later — keep the schema and queries
Postgres-compatible so the move is just a driver swap and a dump/restore. No
SQLite-only features (no `WITHOUT ROWID`, no JSON1-specific ops we can't do in
Postgres, etc.).

## Out of scope for v2

- Billing.
- Team accounts (one user = one business for now).
- Group chats — still 1:1 only, same as v1.
- A customer-facing app. The "app" for the customer is WhatsApp.
