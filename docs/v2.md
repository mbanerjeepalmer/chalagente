# Chalagente v2

v1 was a single-tenant proof: one WhatsApp linked device, one hardcoded script, no
accounts. v2 turns this into a real product: businesses sign up, link their own
WhatsApp number, paste in their info, and an agent fields customer messages on
their behalf.

This doc describes the shipped slice. Anything marked **TODO** is wired up to a
mock or skeleton today and needs the obvious follow-up (usually an API key) to
become real.

## End-to-end story

1. A business lands on the marketing page and signs up with their email.
2. They get a magic link (in dev: printed to the server log via
   `ConsoleMailer`). They click it and land in onboarding.
3. They search Google Maps for their business or type the details in. We save
   what they entered.
4. They paste in free-form extra info — pricing, FAQ, tone. While they're
   there, they can pop open the simulator and try the agent right now, before
   any WhatsApp pairing.
5. They scan a QR to link their WhatsApp number to our agent.
6. We give them a `wa.me/<their-number>` link and QR plus a live transcript
   view so they can send a couple of messages from another phone and watch the
   agent reply. There's a "this works" button that finalises onboarding.
7. From then on, customers WhatsApp the business, the agent answers using the
   business's info. The owner can flip the agent off any time from the
   dashboard.

## Layout

```

auth_adapter.go      bridges internal/store ↔ internal/auth

onboarding.go        4-step wizard (business / extra / whatsapp / test)
dashboard.go         /app — conversations, agent toggle, info, share

web.go               landing page + Mux() + serveHTTP

internal/store/      multi-tenant data layer (sqlite, Postgres-ready)
internal/auth/       magic-link signup, sessions, middleware
internal/wamanager/  per-tenant whatsmeow client registry

internal/voice/      Provider interface + MockProvider + ElevenLabs skeleton + TTS cache

```

## Auth

Email + magic link, implemented directly in `internal/auth`. No phone
verification — the WhatsApp pairing step is the real proof of control over the
number. No `better-auth` dependency — it's a JS library and the app is Go;
rolling the few hundred lines ourselves was less work than adding a Node
process.

- Tokens are 32 random bytes, base64url, stored as SHA-256 hash. 15 min TTL,
  single use.
- Sessions are 30 days, cookie `chala_session`, HttpOnly + SameSite=Lax.
  `Secure` flag is opt-in via `COOKIE_SECURE=true` so dev over HTTP works.
- `Mailer` is an interface. `ConsoleMailer` is the dev default and logs
  the URL via `log.Printf`. Plug in a real provider by writing a 10-line
  `SendMagicLink` impl.

## Onboarding flow

Each step saves on submit; users can drop off and come back. The
`/onboarding` router redirects to the right step based on what's filled in.

1. **Business basics** (`/onboarding/business`). Two ways to populate:
   - Search bar runs `maps.Client.Search` — the mock returns canned Mexico-
     themed places that match by substring. Pick a result, the form is
     pre-filled.
   - Type the fields in manually.
2. **Extra info** (`/onboarding/extra`). One big textarea. Also surfaces a
   prominent "open the simulator" CTA so the owner can try the agent the
   moment they have something for it to talk about.
3. **Link WhatsApp** (`/onboarding/whatsapp`). Click a button → background
   call to `WAManager.StartPairing` → poll `/onboarding/whatsapp/status` →
   render the live QR PNG. When `success` arrives we persist the device JID
   on the business and redirect to step 4.
4. **Test run** (`/onboarding/test`). Shows the `wa.me` link, the live SSE
   transcript, and a "Listo, esto funciona" button that drops the owner into
   `/app`.

## Demo mode

`/app/demo`: WhatsApp-style in-app chat that runs the full agent + voice
pipeline without WhatsApp. Available the moment a business has a name —
i.e. **before** WhatsApp pairing — so owners can experience the agent during
onboarding.

- Text: typed in, posted to `/app/demo/send`, agent reply rendered as a
  bubble.
- Voice: file-upload (mic recording UI TODO). Audio bytes go through
  `Voice.Transcribe`, then the transcript goes to the agent, then the reply
  goes through `Voice.Synthesize` and gets attached as a playable `<audio>`
  bubble.
- History is in-memory per business (`demoHistory` on `App`). A "Reiniciar
  chat" button clears it. Persisting demo conversations would muddy the real
  conversation log, so we don't.

## Admin dashboard

One page per business, at `/app`. Tabs:

- **Conversations** (`/app`). Most recent threads with the last line of each,
  plus a live SSE feed of incoming/outgoing messages.
- **Información** (`/app/business`). Same fields as onboarding, editable.
- **Probar agente** (`/app/demo`). The simulator described above — always
  available from the dashboard, not just during onboarding.
- **Share card** (sidebar of the main tab). Customer-facing QR + `wa.me`
  link, PNG served from `/app/qr.png`.
- **Agent on/off toggle**. Big button on the main tab; when off, inbound
  messages are still logged but the agent doesn't reply.

## Agent

`internal/agent.Engine` interface; one impl today (`MockEngine`).

- `BuildSystemPrompt(BusinessContext) string` produces Spanish prose that
  includes name, address, hours, extra info, today's date, behaviour rules.
  Omits empty fields cleanly.
- `Request` carries `SystemPrompt`, `History`, `Incoming`, and `Business`.
  Real LLM engines will only need `SystemPrompt` + `History` + `Incoming`;
  `Business` exists so `MockEngine` can answer questions about the current
  tenant without parsing prose.
- `MockEngine` has six deterministic rules (hours, location, price, audio-no-
  text, image, default) so tests can assert on outputs and dev sessions feel
  alive without any LLM credit burn.

### TODO

- Real LLM engine. The interface is set up to swap in a Claude- or OpenAI-
  backed impl: take `SystemPrompt`, send it as the system message; map
  `History` to user/assistant turns; send `Incoming`. Attachments need
  multimodal models. Probably ~150 lines plus a key.
- **Tool calling.** `Reply` is just `{Text string}` today. Adding
  `ToolCalls []ToolCall` and a tool-result round-trip is the path to custom
  webhook tools.

## ElevenLabs (interface layer, not the agent's problem)

Two pieces, both bypass the agent's tool-calling — they run unconditionally on
the message envelope before/after the agent sees it. Today: mock provider
makes both work end-to-end without keys. Real provider: skeleton in
`internal/voice/elevenlabs.go`, returns `ErrNoAPIKey` until configured.

**Inbound voice → text.** When a WhatsApp `audioMessage` comes in:

1. Download the encrypted `.ogg` (Opus) via whatsmeow. **TODO** — the pipeline
   currently passes a `[nota de voz recibida]` placeholder instead of real
   bytes. Wire `client.Download` in.
2. POST to `/v1/speech-to-text` with `model_id=scribe_v1`, language
   detection. Skeleton implemented and tested against `httptest.NewServer`.
3. Feed the transcript to the agent as the user turn, prefixed with
   `[voice note, lang=<detected>]`.
4. Stash the detected language on the conversation row
   (`conversations.detected_lang`) so outbound can match.

**Outbound text → voice.** If the inbound was a voice note, reply as a voice
note in the same language. Otherwise text. (Owner can force `always` voice
or `never` via `businesses.voice_mode`.)

1. Take the agent's text output.
2. POST to `/v1/text-to-speech/<voice_id>` with `model_id=eleven_multilingual_v2`,
   output format `mp3_44100_128`. Done in the skeleton.
3. Voice per business (`businesses.voice_id`); default empty → real provider
   needs a sensible fallback voice ID when we configure it.
4. Send via whatsmeow as `audioMessage` with `PTT=true`. **TODO** — the
   demo serves the bytes back over HTTP for playback; the WA send path
   currently only does text.

`CachedProvider` wraps any `Provider` with FIFO cache keyed on
`SHA-256(voiceID + text)`. Same text doesn't re-bill ElevenLabs.

Errors fall back to text: STT fail → "no pudimos escucharte"; TTS fail →
send text reply. Single `ELEVENLABS_API_KEY` env var for now.

## Multimodal

Images and short videos from customers reach `processIncoming` and get a kind
of `image` / `video` in the stored message. The agent gets an `Attachment` of
that kind with no payload yet. **TODO** — download the media via
`client.Download` and pass bytes (or a fetchable URL) when we move to a
vision-capable LLM.

Outbound is text or voice only — no generated images.

## Storage

Single shared SQLite file (`/data/app.db`). Two table namespaces in the same
file:

- App tables (`users`, `sessions`, `magic_link_tokens`, `businesses`,
  `conversations`, `messages`, `tool_configs`, `schema_migrations`) owned by
  `internal/store`.
- Whatsmeow tables (`whatsmeow_*`) owned by `sqlstore`. Both processes hold
  separate `*sql.DB` handles on the same file with
  `_busy_timeout=5000&_fk=1` and `SetMaxOpenConns(1)` on our side.

Multi-tenant is by row, not by file: `businesses.user_id` and
`businesses.wa_device_jid` map an owner and a whatsmeow device to a tenant.
`WAManager` runs one `whatsmeow.Client` per business, all reading from the
same store.

SQLite for now, Postgres later — the schema is portable. We avoid:

- `INTEGER PRIMARY KEY AUTOINCREMENT` (use TEXT UUIDs)
- `WITHOUT ROWID`
- SQLite-only JSON1 functions (we store JSON as TEXT and parse in Go)
- `ON CONFLICT … DO UPDATE` (we do read-then-update-or-insert)

There's one `?`-style placeholder TODO flagged in `internal/store/migrations.go`
for the Postgres swap (`$1, $2` instead of `?`); a tiny driver-aware helper
fixes it when we do the move.

## WhatsApp manager

`internal/wamanager.Manager`:

- Holds a `map[businessID]*whatsmeow.Client` plus a reverse `map[jid]→businessID`.
- `StartPairing(ctx, businessID)` calls `container.NewDevice()`, creates a
  client, attaches an event handler, returns the QR channel. On `success` we
  capture the device JID and register it in the reverse map.
- `StartPaired(ctx, businessID, jid)` connects an existing device on boot.
- `SetEventHandler(fn)` — single callback receives `(businessID, evt)`.
  `pipeline.go` does all the message routing logic.
- `Shutdown()` on SIGTERM.

Tests cover registry semantics. The network-coupled paths (`StartPairing`,
`StartPaired`) are exercised by real use rather than mocked, because mocking
the whatsmeow `Client` is more code than it's worth.

On boot, `main.go` runs `bootPairedTenants` which scans
`businesses.wa_device_jid IS NOT NULL` and reconnects each tenant.

## Configuration

Env vars (all optional):

- `DB_PATH` — sqlite path. Default `./data/app.db`.
- `HTTP_ADDR` — listen address. Default `:8080`.
- `BASE_URL` — used to build magic-link URLs. Default `http://localhost:8080`.
- `COOKIE_SECURE` — `true` to set Secure on the session cookie. Default off.
- `ELEVENLABS_API_KEY` — currently unread (mock provider). Adding a real
  provider in `main.go` flips this on.
- `GOOGLE_PLACES_API_KEY` — same story for Maps.

## Out of scope for v2

- Billing.
- Team accounts (one user = one business for now).
- Group chats — still 1:1 only, same as v1.
- A customer-facing app. The "app" for the customer is WhatsApp.
